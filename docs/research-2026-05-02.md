# Todobar Research Notes - 2026-05-02

This document captures the current research pass for turning Todobar from a web
preview into a serious cross-platform desktop tool.

## Product Thesis

Todobar should be a native-feeling right-edge task surface that can appear over
any app, take input quickly, and disappear without stealing the user's flow. The
browser version is only a development preview.

The core product should work without AI:

- open and close instantly
- capture a task from anywhere
- show Today, Inbox, Month Plan, Later
- persist locally
- stay keyboard-first
- feel calm and reliable

AI and MCP should make the tool stronger, but they should not become the source
of truth. They should suggest plans, summarize context, and propose task changes
that the user can approve.

## Runtime Choice

Recommendation: start with Tauri v2.

Why Tauri first:

- smaller and lighter than Electron for an open-source utility
- React UI can be reused directly
- Rust backend is a good fit for local storage, native shortcuts, tray, window
  placement, and future security boundaries
- Tauri's permission system matches the product goal of explicit local
  capabilities

Why Electron stays as fallback:

- Electron has mature window APIs and predictable Chromium rendering
- always-on-top, frameless windows, global shortcuts, tray, auto-update, and
  multi-monitor behavior are heavily used in production Electron apps
- if Tauri window placement or overlay behavior blocks the UX, Electron should
  be reconsidered before over-engineering around the limitation

Decision: implement the UI through a runtime adapter so the app can run in web,
Tauri, or Electron without rewriting the product layer.

## Desktop Capabilities

The first native shell needs:

- one frameless right-edge sidebar window
- one normal settings window
- global shortcut: show/hide sidebar
- global shortcut: quick capture
- tray/menu bar control
- launch at login
- per-monitor placement
- local database
- update channel
- code signing path for real distribution

Tauri supports the required primitives through official APIs and plugins:

- Tauri can be added to an existing Vite/React project, which fits the current
  repo.
- Windows development requires Microsoft C++ Build Tools and WebView2; macOS
  development requires Xcode or Command Line Tools.
- Capabilities are the mechanism for granting frontend windows/plugin access to
  native APIs.
- global shortcut plugin can register and unregister shortcuts such as
  `CommandOrControl+Shift+T`
- window options include `alwaysOnTop`, `decorations`, `focusable`,
  `skipTaskbar`, initial `x/y`, width/height, and monitor metadata
- SQL plugin supports SQLite and migrations
- autostart plugin supports enable, disable, and state checks
- updater plugin signs update artifacts

Known native risks:

- transparent windows on macOS can require private APIs in Tauri, which would
  block Mac App Store acceptance. Avoid transparency as a dependency.
- Windows shadows and undecorated windows have platform-specific behavior.
- multi-monitor and DPI positioning must be tested on real Windows hardware.
- global shortcut registration can fail if another app already owns a shortcut.
  The app must expose editable shortcuts.

## Widgets

Do not make native widgets the first milestone.

Windows widgets are possible through packaged Win32 desktop apps or PWAs, but
they are a separate widget-provider surface, not the same thing as a universal
right-edge overlay. macOS widgets are built with WidgetKit and are also a
separate native extension surface.

Better first version:

- build a high-quality right-edge overlay window
- add tray/menu-bar controls
- later add Windows Widget and macOS WidgetKit support only for glanceable
  Today/Inbox counts

## Storage

Recommended path:

1. Web preview: `localStorage` for seed prototype only.
2. Desktop milestone: SQLite with migrations.
3. Sync later: change log and conflict model after the local task model is
   stable.

Suggested core tables:

- `tasks`
- `lists`
- `projects`
- `task_events`
- `sources`
- `suggestions`
- `mcp_connections`
- `audit_events`

Design rule: every AI-created or externally imported task should carry source
metadata and audit history.

## AI and MCP Architecture

MCP exposes tools, resources, and prompts:

- Tools are model-invoked actions like querying APIs or creating items.
- Resources are application-provided context, such as files, schemas, notes, or
  project state.
- Prompts are reusable workflows or templates.

For Todobar, MCP should be client-side integration infrastructure, not random
buttons wired into the task UI.

Recommended layers:

```text
Task UI
  -> task domain model
  -> suggestion engine
  -> AI provider adapter
  -> MCP client adapter
  -> permission/audit layer
```

First AI actions:

- `plan_today`: turn selected tasks and context into a proposed Today list
- `split_task`: break a vague task into next actions
- `summarize_inbox`: group raw captured tasks into projects
- `extract_tasks`: pull candidate tasks from selected notes/files
- `defer_intelligently`: suggest Later/Month Plan moves

AI provider direction:

- use a provider adapter so OpenAI, local Ollama, and other OpenAI-compatible
  endpoints can be swapped later
- prefer structured outputs for task proposals
- stream long planning responses only if the UI has a clear pending state
- keep API keys out of the browser/client surface
- support local-only AI later for privacy-sensitive users

First MCP connectors:

- local files and Markdown
- GitHub issues and PRs
- calendar context
- Notion or docs
- Linear/Jira-style issue trackers

MCP runtime note:

- MCP stdio servers are launched as subprocesses. Tauri sidecars can bundle and
  run external binaries through the shell plugin, but every sidecar must be
  explicitly configured in capabilities. This is a good fit for local MCP
  connectors if we keep process lifecycle and permissions strict.

Security rules:

- external reads require visible scope
- external writes require confirmation
- tool input/output should be logged
- tool descriptions and resource content are untrusted
- AI suggestions are drafts until approved
- no API keys in client/browser code

## UX Patterns To Borrow

Task products show strong patterns worth adapting:

- Todoist uses global desktop shortcuts for show/hide and Quick Add.
- Things focuses on Quick Entry from any app and low-friction capture.
- Raycast-like command flows suggest a future command palette for actions,
  connectors, and AI workflows.

Todobar should not become a full project-management dashboard. It should be a
thin edge surface:

- capture
- decide what matters today
- see the next few commitments
- ask AI for a plan when helpful
- approve or reject external changes

## Creative Ideas

- Edge Peek: hover the screen edge to reveal only the next three tasks.
- Focus Strip: one task pinned at the top with a timer and "done / defer /
  split" controls.
- AI Inbox Cleaner: local draft grouping of captured tasks by project.
- MCP Context Pills: visible chips showing which sources are currently allowed,
  such as "GitHub: read issues" or "Calendar: today only".
- Command Mode: `Cmd/Ctrl + K` inside the sidebar for "capture", "plan today",
  "open project", "connect source", "review suggestions".
- Meeting Sweep: after a calendar event, suggest follow-up tasks from notes.
- Project Radar: show only stale or blocked tasks from connected tools.
- Privacy Ledger: a small audit view of every external read/write and AI
  suggestion.
- Local Review Queue: AI suggestions appear as approve/reject cards, not silent
  edits.
- Portable Data: export/import as JSON and Markdown from day one.

## Open Source License Direction

Recommendation: Apache-2.0 as the default public license. It is permissive like
MIT, but includes explicit patent-license language, which is useful for a tool
that may attract contributors, plugin authors, and downstream commercial users.

MIT remains viable if the goal is maximum simplicity. AGPL should be reserved
for a future scenario where hosted sync becomes central and we intentionally
want network-use source sharing.
